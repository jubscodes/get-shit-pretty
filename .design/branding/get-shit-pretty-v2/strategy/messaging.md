# Messaging

> Phase: strategy | Brand: get-shit-pretty | Generated: 2026-03-19

## Core Message

**Design decisions, not design debt.**

GSP walks builders through brand-to-build inside the terminal -- so that every color, every type choice, every spacing decision is intentional, coherent, and yours.

This is the single idea every other message supports. It reframes the problem (design debt from speed) and the solution (intentional decisions through guidance) in language both personas understand.

## Supporting Messages

### 1. "Coherent design shouldn't require a different tool."

**Proof points:**
- GSP runs inside Claude Code, OpenCode, Gemini CLI, and Codex -- the same terminals where builders already work
- The full brand-to-build journey (strategy through design system) happens without leaving the CLI
- Output is design tokens, component foundations, and brand documentation -- artifacts that live in the codebase, not a separate app

**Persona alignment:** Speaks directly to Dev's frustration that design feels like a separate discipline requiring separate tools. Also addresses Des's frustration that handoff culture is dying -- GSP puts design thinking where code happens.

### 2. "Pretty is the outcome. Design that works is the goal."

**Proof points:**
- GSP's process starts with strategy (positioning, archetype, messaging) before any visual decision is made
- Every visual recommendation comes with rationale -- why this color, why this scale, why this spacing
- The dual-diamond methodology ensures brand thinking precedes component building

**Persona alignment:** Reframes the name for both audiences. For Dev, it signals that GSP is not about decoration -- it is about products that communicate clearly. For Des, it validates that GSP respects the design process, not just the output.

### 3. "Your product works. Does it look like it works?"

**Proof points:**
- AI-assisted builders ship 98% more code but with inconsistent UIs and gut-call design decisions
- Users trust products that look considered -- coherent spacing, intentional color, readable type signal competence
- GSP's design system output produces tokens and components that maintain coherence across the product

**Persona alignment:** Hits Dev's aspiration directly -- they want to be proud of the whole product, not just the code. For Des, it articulates the value of their discipline in builder language.

## Elevator Pitch

GSP is a design process for your terminal — like a professional design engineer joined your team. You run a command, and it walks you through brand-to-build: positioning, color rationale, type scales, component foundations. Everything stays in your codebase. No Figma, no handoff, no gut calls. The difference between shipping code and shipping a product is a design process. That's what this is.

*30-second version:*

GSP puts a design engineer on your team, inside the terminal. Brand strategy to design tokens to components — real process, all in the CLI, all in your codebase. The difference between shipping code and shipping a product is a design process. That's what this is.

## Tagline Directions

### Direction 1: "Design that ships with the code."

Emphasizes integration -- design is not a separate deliverable, it ships alongside the product, in the same environment, owned by the same builder. Functional, clear, speaks to Dev first.

### Direction 2: "Walk through it. Build it. Ship it."

Emphasizes the journey and the Guide archetype. Three beats mirror GSP's process without jargon. Warm, active, speaks to both personas.

### Direction 3: "Your terminal. Your brand. Built together."

Emphasizes ownership and environment. The builder keeps control; GSP is the companion, not the authority. Speaks to the value of terminal-native and builder ownership simultaneously.

## Audience-Segment Mapping

### Dev (Primary Persona: The Builder Who Ships Code, Not Product)

| Element | Dev-specific framing |
|---------|---------------------|
| **Core frustration addressed** | "I know my product looks inconsistent but I don't know how to fix it without leaving my terminal." |
| **Opening line** | "Your product works. Here's how to make it look like it works." |
| **Value prop** | Design guidance in the same CLI where you already build. No context switch. No design tool learning curve. |
| **Trust signal** | GSP's own identity was built with GSP. Open source. Terminal-native. Opinionated but not prescriptive. |
| **Channel voice** | GitHub README, dev Twitter/Bluesky, HN comments: technical credibility first, then warmth. Lead with what it does, follow with how it feels. |
| **Key phrases** | "design process," "design engineer on your team," "brand-to-build," "inside your terminal," "design tokens in your codebase," "shipping a product, not just code" |
| **Avoid** | Design jargon without context. "Visual hierarchy" without showing what that means. Anything that sounds like a design agency pitch. |

### Des (Secondary Persona: The Designer Disconnected From the New Era)

| Element | Des-specific framing |
|---------|---------------------|
| **Core frustration addressed** | "Developers are shipping entire products with AI and my handoffs are gathering dust. I need to be where the building happens." |
| **Opening line** | "Your design process, where products are actually built now." |
| **Value prop** | GSP brings design thinking into the terminal -- real process, real methodology, not a "make it pretty" button. Designers who learn GSP bridge the gap to where products ship. |
| **Trust signal** | Dual-diamond methodology. Strategy before visuals. Archetype-driven identity. The process respects design craft. |
| **Channel voice** | Design Twitter/Bluesky, Figma community, design engineering blogs: lead with process and craft, then show the terminal as an exciting new canvas. |
| **Key phrases** | "design process in the terminal," "dual-diamond methodology," "strategy to system," "design engineering," "build, not just design" |
| **Avoid** | Implying designers are obsolete. Framing the terminal as replacing Figma. Anything condescending about traditional design workflows. |

### Shared (Both Personas as "Builders")

| Element | Shared framing |
|---------|---------------|
| **Unifying identity** | Builders -- people who make things and care about what they make. |
| **Shared aspiration** | Ship products where the design is as intentional as the engineering. |
| **Shared language** | "Craft," "coherent," "intentional," "ship," "build." |
| **Community tone** | Inclusive expertise. Everyone is learning. The terminal is common ground. |
